## 1. The mechanism (Lua)
- **`tx_with_retry(bytes, opts)`** (dv:3599-3639): if `RETRY_ELIGIBLE[label]` → `tx_stash[label] = {bytes, opts,
  retries_left=TX_DEFER_MAX_RETRIES(3)}`; duty pre-check (`check_duty_cycle` → `duty_cycle_blocked` + defer via
  after, draw-free); `self:tx`. `RETRY_ELIGIBLE = {CTS, CTS-dup, DATA, ACK, K-dup, NACK, Q}` (dv:3073) — **RTS is
  NOT eligible** (its retry is the rts_timeout, not the stash).
- **`on_radio_busy(info)`** (dv:12081-12215): the sim fires it when a handed TX hits a busy channel
  (`SimController.cpp:1322/1335`). (a) emit `radio_busy`; (b) `label==RTS && pending_tx` → clear `awaiting_cts` +
  emit `rts_tx_blocked` (no stash retry); (c) `label==DATA && pending_tx` → clear `awaiting_ack` + cancel the
  ack-timeout + emit `data_tx_blocked`; (d) DATA-M broadcast retry (R5 channel plane — DEFER); (e) the **stash
  retry:** `stash = tx_stash[label]`; none → no retry; `retries_left<=0` → `tx_giveup` + clear stash;
  else `retries_left--`; re-issue after `wait + 2 + rand(0, lbt_backoff_ms+1)` [**DRAW**].
- The TX architecture (Lua): `tx_initiating` → (LBT) → `tx_with_retry`; `tx_flood` → `tx` (beacons NOT eligible);
  direct response TXs (CTS/DATA/ACK/handle_data NACKs) → `tx_with_retry`. So **every TX except the beacon routes
  through `tx_with_retry`** (the stash + duty pre-check).

## 2. DETERMINISM (gate-inert; the cross-engine gate is the new verification)
The on_radio_busy retry draws `rand(0, lbt_backoff_ms+1)` — but ONLY when the sim fires `on_radio_busy`, which
needs a handed TX to hit a busy channel. In every current gate `lbt_enabled=false` (the sim's LBT defer is gated
on it) AND duty is healthy AND the firmware half-duplex never self-collides → **on_radio_busy never fires → no
stash-retry draw → R4.5b byte-identical on every gate** (un-stubbing is inert; verify the 5 gates stay band-0).
The `tx_with_retry` stash + duty pre-check are draw-free additions (the duty defer uses a timer, no rand) and
inert at healthy duty. **THE NEW VERIFICATION (R4.5a #01):** a `lbt_enabled=true` scenario with forced
contention — the firmware LBT defers the initiating TXs (R4.5a), the responses (CTS/DATA/ACK) can't LBT-defer so
a busy channel makes the sim fire on_radio_busy → the firmware retries from the stash. Build it as a delivery-band
differential; both engines must align (the on_radio_busy retry draw matches the Lua draw-for-draw).

## 3. The sim-LBT-vs-firmware-LBT reconciliation (the open question R4.5a flagged)
After R4.5a the firmware LBT-defers the INITIATING TXs past `busy_until` before handing to the radio, so the sim's
LBT net (`SimController:1322`) should rarely fire for them. It DOES fire for the non-LBT response TXs (CTS/DATA/ACK)
when the channel is busy — that's exactly what on_radio_busy is for. **Verify there is no DOUBLE-defer** (firmware
defers, then the sim defers again) for the initiating TXs — if the firmware cleared the channel, the sim net is a
no-op; if a race leaves it busy, on_radio_busy catches it (now that it's un-stubbed). Confirm the LUA engine
behaves the same (its firmware also LBT-defers; the sim net catches the residual + the Lua on_radio_busy retries).

## 6. Gate / tests
- **Unit:** `on_radio_busy` for a DATA → clears awaiting_ack + cancels the ack-timeout + re-issues from the stash;
  the retries_left countdown (3 → giveup); the rand-jitter draw (one per retry); RTS → clears awaiting_cts +
  `rts_tx_blocked`, NO stash retry; a non-eligible label → no retry. Rand-order: zero draws when on_radio_busy
  doesn't fire; one per stash retry.
- **Differential:** the 5 gates stay band-0 (un-stub inert). THE NEW `lbt_enabled=true` contention gate (delivery
  band) — the load-bearing cross-engine LBT verification (R4.5a #01).

## 8. IMPLEMENTATION OUTCOME (2026-06-01) — what shipped + TWO port divergences the reconciliation found

Built as specced: `on_radio_busy` un-stubbed (node.cpp), the central `tx_with_retry`/`retry_stashed` (node_mac.cpp,
`_tx_stash[4]` per-label slots cts=0/data=1/ack=2/nack=3, `kRadioBusyRetryTimerId`[19..22]), the `FrameTag` tag
plumbed firmware→sim (`TxParams.tag`→`PendingTx.tag`→`RadioBusyInfo.tag`→`BusyInfo.tag`). **115→118 native + the 5
gates stay band-0 (gate-inert: on_radio_busy never fires at lbt_enabled=false).** Two REAL port divergences surfaced
by running a `lbt_enabled=true` contention scenario and reading the Lua side-by-side — both fixed, both with
regression unit tests:

- **DIVERGENCE 1 — a blocked RTS never retried.** Lua `on_radio_busy(RTS)` clears `awaiting_cts` (dv:12091), but Lua
  `rts_timeout_fire` (dv:6326) does NOT gate on it (it captures `ctr_lo` in the timer closure). OUR
  `rts_timeout_fire` (node_cascade.cpp:208) uses `awaiting_cts` AS the staleness key (the fixed timer id can't carry
  `ctr_lo`), so the original port's literal `awaiting_cts=false` made the already-armed rts_timeout bail → the
  blocked RTS was stranded forever (carol 0/N). **Fix:** on_radio_busy(RTS) does NOT clear `awaiting_cts` (the RTS
  never hit the air, so the node legitimately still awaits CTS; the armed rts_timeout re-RTSes, matching Lua's NET
  behaviour). Every other `awaiting_cts=false` transition cancels `kRtsTimeoutTimerId` first (handle_cts:173,
  handle_nack:389), so the guard stays sound. Test: "a blocked RTS re-RTSes via the already-armed rts_timeout".

- **DIVERGENCE 2 — a DATA retry didn't re-arm the ACK wait.** Lua's DATA `on_handed` (dv:10270-10278) sets
  `awaiting_ack=true` + `start_ack_timeout`, and the stash retry RE-FIRES on_handed (dv:12212). OUR `retry_stashed`
  re-sent the bytes only → the re-sent DATA flew but the sender stayed `!awaiting_ack` with no ack-timeout → the
  returning ACK was dropped + the flight never recovered (latent: DATA blocks are rarer than RTS). **Fix:**
  `retry_stashed` re-arms `awaiting_ack` + `start_ack_timeout` on the DATA slot, guarded by `stash.ctr_lo ==
  pending_tx->ctr_lo` (new `TxStashSlot.ctr_lo`, set in tx_with_retry) — matching the Lua on_handed's `ctr_lo` guard.
  Test: "a DATA retry re-arms the ACK wait".

## 9. THE GATE that shipped — `scenarios/r7_lbt_busy_diff.json` (single-flow neutrality, NOT a contention race)
The spec assumed a "forced-contention" delivery gate where "both engines align draw-for-draw". **That is WRONG for a
2-sender race:** a 2-sender/1-receiver contention is NON-DETERMINISTIC cross-engine — the winner depends on each
engine's distinct rand stream + wire airtime, so it legitimately resolves 2/2 on one engine and 1/2 on the other (a
two-direct-sender variant gave meshroute 2/2 vs lua 1/2, |d|=100pp, NEITHER buggy). So the shipped gate is a SINGLE
deterministic alice→relay→bob flow + lbt_enabled=true + a 900ms beacon `spammer` keeping the channel busy: the
firmware LBT pre-check (tx_lbt_defer) fires in BOTH engines (+ lua hits the sim safety-net → radio_busy on seed 42),
the LBT machinery is ACTIVE not stubbed, yet delivery stays 1/1 (band-0, --min-delivery full) across a 10-seed
sweep. `dm_diff_band:r7_lbt_neutral` in run_tests.sh. The on_radio_busy retry MECHANICS (race-triggered) are
UNIT-tested deterministically — that is the right validation for a callback whose trigger is a same-step race, NOT a
flaky delivery gate. [[feedback_wire_format_multileaf_sweep]] sibling lesson: contention ≠ deterministic.
